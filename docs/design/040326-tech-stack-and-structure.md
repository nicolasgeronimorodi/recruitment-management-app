## 1. Tech Stack

### 1.1 Backend — NestJS

| Library                        | Purpose                                      |
| ------------------------------ | -------------------------------------------- |
| **NestJS**                     | Application framework                        |
| **Prisma**                     | ORM — schema lives in infrastructure layer   |
| **@nestjs/passport**           | Authentication integration                   |
| **passport-jwt**               | JWT strategy for Passport                    |
| **@nestjs/jwt**                | JWT token generation and verification        |
| **bcrypt**                     | Password hashing                             |
| **class-validator**            | DTO validation in presentation layer         |
| **class-transformer**          | DTO transformation and serialization         |
| **multer**                     | File upload handling (CV PDFs)               |
| **minio**                      | MinIO JS SDK — store/retrieve files          |

### 1.2 Frontend — Next.js

| Library                        | Purpose                                      |
| ------------------------------ | -------------------------------------------- |
| **Next.js (App Router)**       | Framework — hybrid SSR + CSR                 |
| **React Context API**          | Client-side state management for backoffice  |
| **React Hook Form**            | Form handling (performant, minimal re-renders)|
| **Zod**                        | Schema validation (type-safe, shareable)     |
| **Tailwind CSS**               | Utility-first CSS framework                  |
| **shadcn/ui**                  | Pre-built components (tables, forms, modals) based on Radix UI + Tailwind |

**Rendering strategy:**
- **Public pages** (JobDescription view for applicants): SSR — fast initial load, no auth required
- **Backoffice pages** (RecruitingOfficer dashboard): CSR — behind auth, Context API for state

### 1.3 Infrastructure

| Service        | Image / Technology | Purpose                          |
| -------------- | ------------------ | -------------------------------- |
| **PostgreSQL** | `postgres:16`      | Relational database              |
| **MinIO**      | `minio/minio`      | Object storage (S3-compatible)   |

### 1.4 Orchestration

| Tool             | Purpose                                           |
| ---------------- | ------------------------------------------------- |
| **Docker**        | Containerization of all services                  |
| **Docker Compose**| Multi-service orchestration for local development |

## 2. Docker Compose Architecture

```
docker-compose.yml
  │
  ├── postgres (postgres:16)
  │     Port: 5432
  │     Volume: postgres_data (persistent)
  │     Seed: countries, fields, initial RecruitingOfficer
  │
  ├── minio (minio/minio)
  │     Ports: 9000 (API), 9001 (Console UI)
  │     Volume: minio_data (persistent)
  │     Init: auto-create bucket for CVs
  │
  ├── backend (build: ./backend)
  │     Port: 3000
  │     Depends on: postgres, minio
  │     Volume: ./backend/src mounted for hot reload
  │     Runs: npx prisma migrate deploy && npm run start:dev
  │
  └── frontend (build: ./frontend)
        Port: 3001
        Depends on: backend
        Volume: ./frontend/src mounted for hot reload
        Runs: npm run dev
```

### 2.1 Database Seeding

Seeding runs as part of the backend startup process via Prisma's seed mechanism (`prisma/seed.ts`).

**Seeded data:**
- **Countries** — lookup table (e.g., Argentina, Chile, Colombia, México, etc.)
- **Fields** — business/industry types (e.g., Technology, Finance, Healthcare, etc.)
- **Initial RecruitingOfficer** — first user account with pre-configured credentials

#### Key Architecture Principles

1. **Domain layer has zero dependencies** — no imports from NestJS, Prisma, or any framework. Pure TypeScript classes.
2. **Repository interfaces (ports)** live in `domain/repositories/`. Implementations (adapters) live in `infrastructure/persistence/`.
3. **Use cases** orchestrate domain logic. Each use case is a single class with one public `execute()` method.
4. **DTOs** belong to the application layer — they validate input before it reaches the domain.
5. **NestJS modules** live in the infrastructure layer — they wire everything together via dependency injection.

### 3.3 Frontend — Feature-based Structure

```
frontend/
├── Dockerfile
├── package.json
├── tsconfig.json
├── next.config.js
│
├── public/                       # static assets
│
└── src/
    ├── app/                      # Next.js App Router
    │   ├── layout.tsx            # root layout
    │   ├── page.tsx              # landing/redirect
    │   │
    │   ├── (public)/             # route group: SSR, no auth
    │   │   └── jobs/
    │   │       └── [id]/
    │   │           ├── page.tsx          # view job description
    │   │           └── apply/
    │   │               └── page.tsx      # application form
    │   │
    │   └── (backoffice)/         # route group: CSR, auth required
    │       ├── layout.tsx        # backoffice layout (sidebar, nav)
    │       ├── dashboard/
    │       │   └── page.tsx
    │       ├── companies/
    │       │   └── page.tsx
    │       ├── job-descriptions/
    │       │   ├── page.tsx              # list
    │       │   ├── new/
    │       │   │   └── page.tsx          # create
    │       │   └── [id]/
    │       │       ├── page.tsx          # detail + edit
    │       │       └── applications/
    │       │           └── page.tsx      # list applications
    │       └── officers/
    │           └── page.tsx              # manage recruiting officers
    │
    ├── contexts/                 # React Context providers
    │   ├── auth.context.tsx      # auth state: user, token, login/logout
    │   └── ui.context.tsx        # UI state: sidebar open, notifications
    │
    ├── hooks/                    # custom hooks
    │   ├── useAuth.ts
    │   └── useApi.ts
    │
    ├── services/                 # API client layer
    │   ├── api.ts                # base axios/fetch config
    │   ├── auth.service.ts
    │   ├── company.service.ts
    │   ├── job-description.service.ts
    │   └── application.service.ts
    │
    ├── components/               # reusable UI components
    │   ├── ui/                   # generic (Button, Input, Modal, etc.)
    │   └── forms/                # form components with React Hook Form + Zod
    │
    ├── lib/                      # utilities
    │   └── validations/          # Zod schemas
    │       ├── auth.schema.ts
    │       ├── company.schema.ts
    │       ├── job-description.schema.ts
    │       └── application.schema.ts
    │
    └── types/                    # TypeScript type definitions
        └── index.ts
```

#### Key Frontend Principles

1. **Route groups** `(public)` and `(backoffice)` separate SSR and CSR rendering strategies without affecting URL paths.
2. **Context API** manages two concerns: **auth state** (token, current user, permissions) and **UI state** (sidebar, notifications). This is where Context API fits naturally — global client-side state that many components need but doesn't warrant a heavy state management library.
3. **Services layer** abstracts API calls — components never call `fetch` directly.
4. **Zod schemas** in `lib/validations/` are used by React Hook Form for client-side validation.
